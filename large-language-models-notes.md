## 1. 大语言模型主线：从“预测下一个词”到“会查资料再回答”【了解】

语言模型的基本目标是给一个词序列分配概率，或者根据前文预测下一个 token：

$$
p(x_1,x_2,\ldots,x_T)=\prod_{t=1}^{T}p(x_t|x_{<t})
$$

早期的统计语言模型用 N-gram 估计短窗口里的词序列概率；RNN 和 Transformer 则把上下文表示成向量，让模型能学习更复杂的语义关系。大语言模型的核心变化不是目标突然变了，而是模型规模、数据规模、训练方式、推理方式和外部工具能力一起变强了。

可以把本章内容串成一条线：

1. 先学会语言建模：根据上下文预测 token。
2. 再用 Transformer 扩展上下文建模能力。
3. 再用大规模预训练让模型吸收通用语言知识。
4. 再用后训练让模型更会按人类意图说话。
5. 最后用 RAG 等外部增强方法，让模型在参数知识之外查资料、降幻觉、降成本。

## 2. 基于统计的语言模型：N-gram【了解】

N-gram 指长度为 `n` 的连续词序列。它的基本想法是：当前词的概率不依赖完整历史，只依赖前面 `n-1` 个词。

例如：

- Unigram：不看上下文，只看一个词本身的出现频率。
- Bigram：看前一个词。
- Trigram：看前两个词。

N-gram 的优点是直观、可统计、可解释，曾经广泛用于输入法、拼写纠错、机器翻译等任务。缺点也很明显：窗口有限，难以捕捉长程依赖；按字面匹配，不能很好处理语义相近但表面不同的表达；`n` 变大后参数量指数增长，而且容易出现没见过的组合导致零概率。

<details>
<summary>展开：N-gram 概率公式</summary>

完整语言模型希望估计：

$$
p(w_1,w_2,\ldots,w_T)=\prod_{t=1}^{T}p(w_t|w_1,\ldots,w_{t-1})
$$

N-gram 做了马尔可夫假设，只保留最近的 `n-1` 个词：

$$
p(w_t|w_1,\ldots,w_{t-1})\approx p(w_t|w_{t-n+1},\ldots,w_{t-1})
$$

用语料频率估计时：

$$
p(w_t|w_{t-n+1},\ldots,w_{t-1})
=
\frac{C(w_{t-n+1},\ldots,w_t)}
{C(w_{t-n+1},\ldots,w_{t-1})}
$$

这里 $C(\cdot)$ 表示某个词序列在语料中出现的次数。`n` 太小，上下文不够；`n` 太大，很多组合没见过，泛化变差，计算和存储也变重。

</details>

## 3. 语言模型采样：模型给概率，人类定策略【了解】

语言模型每一步输出的是整个词表上的概率分布，不是直接输出唯一答案。采样策略决定“从概率分布里选哪个 token”。

| 方法 | 做法 | 特点 |
|---|---|---|
| 贪心搜索 | 每步都选概率最大的 token | 稳定但容易平庸、重复 |
| Beam Search | 保留多个高概率候选序列 | 适合翻译等目标明确任务，但开放式生成容易“废话文学” |
| Top-k | 每步只在概率最高的 `k` 个 token 中采样 | 增加多样性，但固定 `k` 不够自适应 |
| Top-p | 选取累计概率达到阈值 `p` 的候选集合 | 候选集合大小随分布变化，更灵活 |
| Temperature | 调整概率分布的尖锐程度 | 温度低更保守，温度高更多样也更冒险 |

Top-k 和 Top-p 都是在“候选词集合”上做文章；Temperature 是在“概率分布形状”上做文章。实际生成常常组合使用，比如 `top_p=0.9` 加一个适中的 temperature。

<details>
<summary>展开：Temperature 怎么改变随机性</summary>

模型输出 logits $z_i$ 后，原本概率是：

$$
p_i=\frac{\exp(z_i)}{\sum_j\exp(z_j)}
$$

加入温度 $T$ 后：

$$
p_i=\frac{\exp(z_i/T)}{\sum_j\exp(z_j/T)}
$$

当 $T<1$，大的 logit 会被进一步放大，分布更尖，模型更保守。

当 $T>1$，不同 token 的概率差距被拉平，分布更平，生成更随机。

</details>

## 4. 预训练、曝光偏差与扩展法则【了解】

大语言模型的预训练通常使用自监督任务。Decoder-only 模型最典型的是下一词预测：

$$
\max_\theta \sum_t \log p_\theta(x_t|x_{<t})
$$

训练时常用 Teacher Forcing：模型预测第 `t` 个 token 时，前面喂给它的是真实 token，而不是模型自己上一步生成的 token。这样训练稳定、并行友好，但会带来曝光偏差。

曝光偏差的意思是：训练时模型总是在“标准答案前缀”上学习，推理时却必须接着“自己生成的前缀”往下走。一旦前面生成错了，后面会在错误上下文里继续生成，错误可能累积，幻觉风险也会上升。

扩展法则 Scaling Law 讲的是模型性能与参数量、数据量、计算量之间存在经验规律。大模型能力提升的重要原因之一，是模型规模和训练数据规模一起扩大；但课件也强调，规模不是唯一答案，数据质量、训练范式和后训练同样关键。

<details>
<summary>展开：语言模型损失函数</summary>

对 Decoder-only 语言模型，训练样本是 token 序列 $x_1,\ldots,x_T$。模型希望最大化真实下一个 token 的概率：

$$
\max_\theta \sum_{t=1}^{T}\log p_\theta(x_t|x_{<t})
$$

实现中通常最小化负对数似然，也就是交叉熵损失：

$$
L(\theta)=-\sum_{t=1}^{T}\log p_\theta(x_t|x_{<t})
$$

这就是“接龙能力”的数学形式。模型不是直接被训练成“懂道理”，而是通过海量文本接龙学到词法、语法、事实、风格和某些推理模式。

</details>

## 5. Transformer 的两类核心模块【了解】

Transformer 主要由注意力模块和全连接前馈模块组成。

- 注意力模块负责“当前 token 应该参考哪些上下文 token”。
- 前馈网络 FFN/MLP 占据大量参数，常被理解为模型存储和变换知识的重要位置。
- 残差连接让深层网络更容易训练。
- 层归一化让训练更稳定。

注意力的直觉是加权平均：每个 token 先对其他 token 打分，再按照权重汇总信息。

<details>
<summary>展开：Scaled Dot-Product Attention</summary>

注意力有三个角色：

- Query：我现在要问什么。
- Key：每个位置有什么索引特征。
- Value：每个位置真正要被取走的信息。

公式是：

$$
\operatorname{Attention}(Q,K,V)
=
\operatorname{softmax}\left(\frac{QK^T}{\sqrt{d_k}}\right)V
$$

$QK^T$ 计算“每个 Query 和每个 Key 的相关性”；除以 $\sqrt{d_k}$ 是为了防止维度变大后点积过大；softmax 把相关性变成权重；最后乘 $V$ 得到加权汇总后的表示。

自注意力和交叉注意力的公式一样，区别在于 $Q,K,V$ 来自哪里。

</details>

## 6. 三种大语言模型架构总览【重点】

Transformer 衍生出三种主流架构：Encoder-only、Encoder-Decoder、Decoder-only。它们最大的差别不是“名字”，而是谁负责哪些 token、注意力矩阵允许看哪里。

| 架构 | 负责的 token | 注意力特点 | 代表模型 | 更适合 |
|---|---|---|---|---|
| Encoder-only | 输入 token | 双向完全自注意力 | BERT、RoBERTa、ALBERT | 理解、分类、匹配、检索编码 |
| Encoder-Decoder | Encoder 负责输入 token，Decoder 负责输出 token | 输入内部完全注意力，输出内部下三角注意力，输出对输入做交叉注意力 | BART、T5 | 翻译、摘要、结构化生成 |
| Decoder-only | 把提示词和生成内容都放在同一条序列里 | 下三角因果自注意力 | GPT、LLaMA、Qwen 等 | 开放式生成、对话、通用助手 |

你前面问到的关键点非常重要：在 Encoder-Decoder 中，Encoder 和 Decoder 负责的 token 通常不是同一批。

例如翻译：

```text
输入：I love you        <- Encoder 处理这些 token
输出：我 爱 你          <- Decoder 处理这些 token
```

所以图里的注意力矩阵不是同一批 token 内部的一个方块，而是把“输入序列 token”和“输出序列 token”放在一起看。

## 7. Encoder-only：输入 token 之间充分互相理解【重点】

Encoder-only 只使用 Transformer 的编码器部分。它的核心是双向自注意力：每个输入 token 都可以看输入序列中的所有位置。

例如句子：

```text
我 不 喜欢 这 部 电影
```

处理“喜欢”时，模型既能看左边的“我 不”，也能看右边的“这 部 电影”。所以它很适合理解类任务，比如情感分类、文本匹配、实体识别、检索向量编码等。

但 Encoder-only 没有天然的自回归解码器，因此不擅长直接一个 token 一个 token 生成长文本。BERT 类模型可以做填空，但不是最自然的开放式生成架构。

## 8. Decoder-only：只看过去，边生成边扩展上下文【重点】

Decoder-only 只使用 Transformer 的解码器部分，并省略原始 Encoder-Decoder 解码器里的交叉注意力子模块。它的核心是因果自注意力，也叫下三角注意力。

对第 `t` 个 token 来说，它只能看 `1` 到 `t` 的 token，不能看未来 token。这样才符合生成过程：

```text
第 1 步：生成 我
第 2 步：看到 我，再生成 爱
第 3 步：看到 我 爱，再生成 你
```

现代对话模型通常把输入提示词和模型回复拼成一条长序列：

```text
用户问题 token + 助手回答 token
```

然后用同一套下三角注意力建模。提示词在回答之前，所以回答 token 可以看见提示词；提示词 token 不能看见未来回答 token。这也是为什么 Decoder-only 可以在没有独立 Encoder 的情况下做问答和对话。

Decoder-only 的优点是架构简单、扩展性强、非常适合大规模预训练和开放式生成。局限是没有 Encoder 那种对输入的双向深度编码，在某些复杂输入理解或严格源到目标转换任务上不一定最经济。

## 9. Encoder-Decoder：输入理解和输出生成分工【重点】

Encoder-Decoder 同时使用编码器和解码器。它的分工最清楚：

- Encoder：负责输入序列 token，目标是把输入整体理解成上下文表示。
- Decoder：负责输出序列 token，目标是根据已经生成的输出和 Encoder 的输入表示，逐步生成下一个 token。

以翻译为例：

```text
Encoder 看：I love you
Decoder 生成：我 -> 爱 -> 你
```

这里 Decoder 每一步手里有两类信息：

1. 我已经生成了什么。
2. 输入句子里哪些部分和当前要生成的词有关。

第一类来自 Decoder 内部的 masked self-attention。第二类来自 cross-attention。

## 10. 三种注意力：完全、下三角、交叉【重点】

课件图中 Encoder-Decoder 的三种注意力可以这样记：

| 模块 | Query 来自 | Key/Value 来自 | 能看哪里 | 作用 |
|---|---|---|---|---|
| Encoder 自注意力 | Encoder token | Encoder token | 全部输入 token | 输入内部充分理解 |
| Decoder 自注意力 | Decoder token | Decoder token | 已生成的输出 token | 保证自回归生成，不偷看未来 |
| 交叉注意力 | Decoder token | Encoder token | 全部输入 token | 生成当前输出时回看输入 |

“完全注意力”不是说注意力权重平均，也不是说每个 token 一定真的强烈关注所有位置。它的意思是没有 mask 限制，理论上可以访问所有对应位置。

### 10.1 Encoder 内部的完全自注意力

Encoder 处理的是输入 token。输入句子在编码时已经完整给定，所以每个输入 token 可以看见所有输入 token。

```text
输入：I love you
```

编码 `love` 时可以同时看 `I` 和 `you`。这对应图里的蓝色完整方块。

### 10.2 Decoder 内部的下三角自注意力

Decoder 处理的是输出 token。输出是一步步生成的，所以第 `t` 个输出 token 只能看前面已经生成的 token，不能看未来。

```text
输出：我 爱 你
```

生成“爱”时可以看“我”，不能看未来的“你”。这对应图里的绿色下三角。

### 10.3 交叉注意力：Decoder 查询 Encoder

交叉注意力发生在 Decoder 里面，但它看的对象是 Encoder 的输出。

最关键的一句：

> 交叉注意力中，Query 来自 Decoder，Key 和 Value 来自 Encoder。

当 Decoder 要生成“我”时，它会拿当前 Decoder 状态作为 Query，去和 Encoder 中 `I love you` 的每个位置做匹配。它可能发现 `I` 最相关，于是把 `I` 对应的 Value 更多地汇总回来。

当 Decoder 要生成“爱”时，它可能更关注 `love`。

当 Decoder 要生成“你”时，它可能更关注 `you`。

所以“解码器对编码器的完全注意力”的准确理解是：

> Decoder 的每一个输出位置，都可以访问 Encoder 的全部输入位置。

它不是说 Decoder 和 Encoder 负责同一批 token。恰恰相反，它们负责不同 token：Encoder 管输入，Decoder 管输出。交叉注意力就是 Decoder 在生成自己的输出 token 时，向 Encoder 的输入表示发起查询。

<details>
<summary>展开：用矩阵看三种注意力</summary>

假设输入长度为 3，输出长度为 3。把行理解成 Query 的位置，把列理解成 Key/Value 的位置：

$$
\begin{array}{c|cc}
 & \text{Encoder tokens} & \text{Decoder tokens} \\\hline
\text{Encoder queries} & \text{Encoder 完全自注意力} & \text{通常没有这一块} \\
\text{Decoder queries} & \text{交叉注意力} & \text{Decoder 下三角自注意力}
\end{array}
$$

Encoder 自注意力：

$$
Q=X_{\text{enc}}W_Q,\quad K=X_{\text{enc}}W_K,\quad V=X_{\text{enc}}W_V
$$

Decoder 自注意力：

$$
Q=Y_{\text{dec}}W_Q,\quad K=Y_{\text{dec}}W_K,\quad V=Y_{\text{dec}}W_V
$$

交叉注意力：

$$
Q=Y_{\text{dec}}W_Q,\quad K=H_{\text{enc}}W_K,\quad V=H_{\text{enc}}W_V
$$

其中 $H_{\text{enc}}$ 是 Encoder 输出的输入序列表示，$Y_{\text{dec}}$ 是 Decoder 当前层里的输出序列表示。交叉注意力的本质就是“输出位置向输入位置提问”。

</details>

## 11. 后训练：让会接龙的模型更会说话【了解】

只做下一词预测，模型学到的是“文本接龙”。这不等于它会遵循指令、拒绝不合适请求、给出有帮助且诚实的回答。

后训练一般包括：

- 指令微调 SFT：用“用户指令 - 优质回答”数据做监督学习，让模型学会对话格式和任务执行方式。
- 价值对齐：让模型更符合有用、诚实、无害的目标。
- RLHF：用人类偏好训练奖励模型，再用强化学习优化语言模型。
- GRPO 等方法：通过组内相对奖励等方式强化模型的推理和答案质量。

一句话区分：预训练让模型“有知识、会接龙”；指令微调让模型“会按要求回答”；偏好优化和强化学习让模型“更符合人类偏好和任务目标”。

<details>
<summary>展开：RLHF 和 GRPO 的直觉</summary>

RLHF 常见流程：

1. 收集人类偏好数据：同一个问题给多个回答，人类排序。
2. 训练奖励模型：让奖励模型学会判断哪个回答更好。
3. 强化学习优化语言模型：让模型生成更高奖励的回答，同时不要偏离原模型太远。

可以抽象成：

$$
\max_\theta E_{y\sim \pi_\theta(\cdot|x)}[R(x,y)]
-\beta KL(\pi_\theta || \pi_{\text{ref}})
$$

其中 $R(x,y)$ 是奖励，$KL$ 项用于限制模型不要为了追奖励而跑偏。

GRPO 的直觉是：对同一道题采样一组答案，在组内比较相对好坏，用相对奖励推动模型改进。它减少了对单独价值函数/批判模型的依赖，更强调“干中学”和组内相对评价。

</details>

## 12. 参数高效微调：Adapter 与 LoRA【了解】

全量微调要更新全部参数，显存、计算和存储开销都很大。参数高效微调 PEFT 的目标是：冻结大部分预训练参数，只训练很少的新增参数或部分参数。

Adapter-tuning 的做法是在注意力层或前馈层附近插入小型适配器模块。原模型参数基本冻结，只训练 Adapter、层归一化或任务头等少量参数。它的优势是任务适配灵活，可以为不同任务保存不同 Adapter。

LoRA 的核心是低秩适配。它不直接更新完整权重矩阵 $W$，而是学习一个低秩更新量：

$$
W'=W+\Delta W,\qquad \Delta W=BA
$$

其中 $A$ 和 $B$ 是两个小矩阵，秩 $r$ 很小，所以可训练参数远少于全量微调。LoRA 常加在 Attention 或 FFN 的投影矩阵上。推理时可以把 $\Delta W$ 合并回原权重，不额外增加推理计算。

<details>
<summary>展开：为什么 LoRA 能省参数</summary>

假设原始权重矩阵：

$$
W\in R^{d\times k}
$$

全量微调需要学习一个同样大小的更新：

$$
\Delta W\in R^{d\times k}
$$

参数量是 $d\times k$。

LoRA 假设更新矩阵可以近似为低秩分解：

$$
\Delta W=BA
$$

其中：

$$
B\in R^{d\times r},\quad A\in R^{r\times k},\quad r\ll \min(d,k)
$$

可训练参数量变成：

$$
d\times r+r\times k=r(d+k)
$$

当 $r$ 很小时，参数量大幅下降。初始化时常令 $B=0$，让 $\Delta W=0$，这样微调开始时模型行为与原模型一致，训练更稳定。

</details>

## 13. 检索增强生成 RAG：为什么需要外部知识【重点】

RAG 的全称是 Retrieval-Augmented Generation，检索增强生成。它的核心思想是：

> 模型回答前，先从外部知识库检索相关资料，再把资料和问题一起交给生成模型。

这样做主要是为了解决大语言模型的几个问题：

- 参数知识会过时：模型训练后发生的新事实不在参数里。
- 长尾知识记不住：冷门知识很难靠预训练参数完整覆盖。
- 幻觉风险：模型可能在没有可靠依据时编出看似合理的答案。
- 成本问题：与其把所有知识都塞进参数，不如把可检索知识放在外部库里。

RAG 不是让模型变成数据库，而是把“查资料”和“组织语言”拆开：检索器负责找材料，生成器负责读材料并回答。

## 14. RAG 基本组成：知识检索 + 生成增强【重点】

一个基本 RAG 系统包含两大模块：

1. 知识检索：把用户问题编码成查询，从知识库中找相关文档。
2. 生成增强：把检索到的文档和用户问题一起输入大模型，让模型基于外部知识生成答案。

典型流程是：

```text
用户问题
-> 查询构建
-> 检索器召回相关文档
-> 重排器筛选更相关文档
-> 拼接进 Prompt 或注入模型中间层
-> 生成模型回答
```

最常见、最容易实现的是输入端增强：把检索到的文档直接拼到 Prompt 里。这类方法不需要改模型参数，适合调用闭源模型或快速搭建系统。

## 15. RAG 中的编码器：把问题和文档变成可比较的向量【重点】

RAG 里的“编码器”通常指检索器中的文本编码模型。它和 Encoder-Decoder 里的 Encoder 有联系，但不完全是同一个语境。

在稠密检索中，编码器把 query 和 doc 映射成向量：

$$
u=\operatorname{Encoder}(\text{query}),\qquad
v=\operatorname{Encoder}(\text{doc})
$$

然后用相似度判断相关性：

$$
\operatorname{score}(q,d)=u^Tv
$$

常见检索器：

| 类型 | 做法 | 优点 | 局限 |
|---|---|---|---|
| 稀疏检索 | 根据关键词、词频等匹配 | 快、可解释、适合精确词 | 语义泛化弱 |
| 双编码器 | query 和 doc 分别编码成向量再算相似度 | 文档可离线编码，检索快 | query-doc 交互较浅 |
| 交叉编码器 | query 和 doc 拼在一起编码并打分 | 语义交互深，排序准 | 成本高，适合重排 |
| 图检索器 | 利用实体和关系图谱检索 | 适合结构化知识 | 构建维护成本高 |

实际系统常用“双编码器召回 + 交叉编码器重排”：先快，再准。

## 16. RAG 架构分类：黑盒增强与白盒增强【重点】

RAG 可以按是否更新大语言模型参数分成黑盒增强和白盒增强。

| 架构 | 参数是否更新 | 代表思路 | 优点 | 缺点 |
|---|---|---|---|---|
| 黑盒无微调 | 检索器和 LLM 都不更新 | In-Context RALM，把文档放进上下文 | 实现简单、成本低、与模型解耦 | 检索器和 LLM 缺少深度适配 |
| 黑盒检索器微调 | LLM 不变，只调检索器 | REPLUG，用 LLM 反馈训练检索器 | 不改 LLM，成本较低 | LLM 固定，适配上限受限 |
| 白盒仅微调 LLM | 检索器不变，调 LLM | RETRO，把检索信息注入隐藏层 | LLM 更会利用外部信息 | 训练成本高，依赖检索器质量 |
| 白盒协同微调 | 检索器和 LLM 一起调 | ATLAS，用联合目标优化 | 深度交互，效果潜力大 | 实现复杂、资源需求高 |

复习时先记住：黑盒偏工程落地，白盒偏模型内部融合；无微调最简单，协同微调最强但最贵。

## 17. 何时增强、何处增强、何以增强【重点】

课件里 RAG 的重点可以归纳成三个问题。

### 17.1 何时增强

不是所有问题都需要检索。模型本来就会、且答案稳定的问题，检索可能只会增加延迟和噪声。需要检索的场景通常包括：

- 涉及新知识、新政策、新数据。
- 涉及专业文档、企业内部知识库、课程资料。
- 问题很长或很具体，模型参数知识不可靠。
- 需要引用依据或可追溯来源。

“何时增强”的本质是判断：当前问题是否需要外部知识才能可靠回答。

### 17.2 何处增强

外部知识可以在不同位置进入模型：

| 位置 | 做法 | 代表直觉 |
|---|---|---|
| 输入端 | 把检索文档拼进 Prompt | 最常用、最易落地 |
| 中间层 | 将外部知识编码成向量，通过交叉注意力注入隐藏状态 | 更深度融合，但要改模型 |
| 输出端 | 生成后用外部知识校验、修正或引用补全 | 适合事实核查和后处理 |

这里的“中间层增强”又会和前面交叉注意力呼应：模型生成时不只看自己的上下文，还通过交叉注意力读取外部知识向量。

### 17.3 何以增强

也就是用什么外部知识增强：

- 原始文档片段：最常见。
- 结构化表格：适合精确数值和业务数据。
- 知识图谱：适合实体关系。
- 多模态资料：图像、视频、语音等。
- 历史对话和用户画像：适合个性化回答，但要注意隐私和安全。

## 18. 知识检索流程：构库、构问、召回、重排【重点】

RAG 的生成质量很大程度取决于检索质量。检索部分通常包含四步：

1. 知识库构建：采集、清洗、切分、向量化、建索引。
2. 查询构建：改写问题、拆分复杂问题、补充背景。
3. 文本检索：用稀疏检索或稠密检索召回候选文档。
4. 结果重排：用交叉编码器等更强模型重新排序，保留最相关片段。

文档分块很关键。块太大，检索不精准，也容易塞满上下文；块太小，语义不完整。实际常常需要根据文档类型设置 chunk size、overlap、标题层级和元数据。

## 19. RAG 的降本增效【重点】

RAG 的“降本增效”体现在两方面：

- 降本：不必每次为了新知识重新训练大模型，也不必把所有知识都塞进参数；更新知识库比重新训练模型便宜得多。
- 增效：检索到的证据能缩小模型搜索空间，提高事实性、可追溯性和领域适配能力。

但 RAG 不是万能药。它会引入新的误差来源：

- 检索不到正确文档，模型仍可能答错。
- 检索到噪声文档，模型可能被误导。
- 文档太多或太长，模型可能抓不住重点。
- Prompt 拼接不当，会造成上下文冲突。

所以 RAG 系统要同时优化检索质量、重排质量、上下文组织和生成约束。

## 20. 一页速记

- N-gram：用短窗口统计词序列概率，简单但捕捉不了长程依赖。
- 采样：模型给概率，策略决定选哪个 token；Top-k 固定候选数，Top-p 固定累计概率，Temperature 调随机性。
- 曝光偏差：训练时看真实前缀，推理时看自己生成的前缀，错误会累积。
- Encoder-only：输入 token 双向完全注意力，强理解，弱开放生成。
- Decoder-only：因果下三角注意力，强生成，现代通用 LLM 主流。
- Encoder-Decoder：Encoder 管输入，Decoder 管输出，交叉注意力让输出 token 查询全部输入 token。
- 交叉注意力：Query 来自 Decoder，Key/Value 来自 Encoder。
- 后训练：SFT 让模型会按指令说话，RLHF/GRPO 让模型更符合偏好或推理目标。
- Adapter：往模型层里插小模块，只训练小模块。
- LoRA：用低秩矩阵学习权重更新，省显存、易合并。
- RAG：先检索外部知识，再生成答案，重点解决知识过时、长尾知识和幻觉。
- RAG 编码器：把 query/doc 编成向量；双编码器快，交叉编码器准。
- 何时增强：模型参数知识不够可靠时增强。
- 何处增强：输入端最常见，中间层最深度，输出端适合校验修正。
